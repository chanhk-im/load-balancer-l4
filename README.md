# TCP 로드 밸런서

C언어와 raw socket으로 구현한 L4 TCP 로드 밸런서입니다. 라운드 로빈, 최소 연결, 리소스 기반 세 가지 알고리즘을 지원합니다.

## 아키텍처

```
[클라이언트]
     │
     ▼
[로드 밸런서]  ←──(리소스 정보)──  [서버 1]
     │         ←──(리소스 정보)──  [서버 2]
     ├────────────────────────────▶ [서버 1]
     └────────────────────────────▶ [서버 2]
```

- **로드 밸런서**: raw socket으로 TCP 패킷을 캡처하고 IP/TCP 헤더를 재작성(NAT)해 백엔드 서버로 포워딩합니다.
- **서버**: 에코 서버로, 주기적으로 CPU 사용률, 메모리, 연결 수 등 리소스 정보를 LB에 전송합니다.
- **클라이언트**: LB의 IP와 포트로 접속하는 표준 TCP 클라이언트입니다.

### 주요 파일

| 파일                  | 설명                                                  |
| --------------------- | ----------------------------------------------------- |
| `lb.c`                | 로드 밸런서 본체 — 패킷 캡처, NAT 테이블 관리, 포워딩 |
| `server_pool.c/h`     | 백엔드 서버 목록 관리                                 |
| `nat_table.c/h`       | NAT 매핑 테이블 (클라이언트 ↔ lb_port ↔ 서버)         |
| `server_type.h`       | 공유 타입 정의                                        |
| `server/echo_serv.c`  | 백엔드 에코 서버                                      |
| `client/tcp_client.c` | 테스트 클라이언트                                     |

## 요구사항

- Linux (raw socket 및 `/proc` 파일시스템 필요)
- GCC
- 루트 권한 (raw socket)
- `iptables`

## 빌드

```bash
# 로드 밸런서 빌드
make

# 서버 빌드
cd server && make

# 클라이언트 빌드
cd client && make
```

## 실행

### 1. 로드 밸런서 (LB 머신)

iptables RST 패킷 드롭 설정이 필수입니다. 설정하지 않으면 커널이 포워딩 중인 연결에 RST를 보내 연결이 끊어집니다.

```bash
sudo iptables -A OUTPUT -p tcp --tcp-flags RST RST -j DROP
sudo ./lb <LB_IP> <LB_PORT>
```

### 2. 서버 (각 서버 머신)

```bash
./serv <SERV_PORT> <LB_IP> <LB_PORT>
```

LB가 먼저 실행된 상태에서 서버들이 접속해야 합니다. `SERVER_COUNT`만큼 서버가 접속해야 LB가 클라이언트를 받기 시작합니다.

### 3. 클라이언트

```bash
./clnt <LB_IP> <LB_PORT>
```

### 실행 예시

```bash
# LB (192.168.0.10)
sudo iptables -A OUTPUT -p tcp --tcp-flags RST RST -j DROP
sudo ./lb 192.168.0.10 8080

# 서버 1 (192.168.0.11)
./serv 9001 192.168.0.10 8080

# 서버 2 (192.168.0.12)
./serv 9001 192.168.0.10 8080

# 클라이언트
./clnt 192.168.0.10 8080
```

## 로드 밸런싱 알고리즘

`lb.c` 상단의 `ALGORITHM_FLAG`를 변경해 알고리즘을 선택합니다.

```c
#define ALGORITHM_FLAG 1  // 0: 라운드 로빈, 1: 최소 연결, 2: 리소스 기반
```

### 0 — 라운드 로빈 (Round Robin)

요청이 들어올 때마다 서버를 순서대로 돌아가며 선택합니다. 구현이 단순하고 서버 스펙이 동일한 환경에 적합합니다.

### 1 — 최소 연결 (Least Connection) `기본값`

현재 연결된 클라이언트 수가 가장 적은 서버로 새 연결을 보냅니다. 각 서버는 5초마다 현재 연결 수를 LB에 전송하고, LB는 이 값을 기준으로 서버를 선택합니다.

### 2 — 리소스 기반 (Resource Based)

CPU 사용률과 남은 메모리를 합산해 부하가 가장 낮은 서버를 선택합니다. 서버가 `/proc/stat`, `/proc/meminfo`를 읽어 실시간 리소스 정보를 LB에 전송합니다.

```
점수 = cpu_usage + memory_left   (낮을수록 선택)
```

## 동작 원리

LB는 raw socket으로 모든 TCP 패킷을 캡처하고 아래 흐름으로 동작합니다.

```
클라이언트 → LB
  1. NAT 테이블에서 해당 클라이언트 검색
  2. 없으면 알고리즘으로 서버 선택 → NAT 테이블에 등록
  3. 패킷 헤더 수정 (출발지: LB IP/포트, 목적지: 서버 IP/포트)
  4. 체크섬 재계산 후 포워딩

서버 → 클라이언트
  1. 패킷 출처가 서버 풀에 있는지 확인
  2. NAT 테이블에서 lb_port로 클라이언트 검색
  3. 패킷 헤더 수정 (출발지: LB IP/포트, 목적지: 클라이언트 IP/포트)
  4. 체크섬 재계산 후 포워딩
```

## 주의사항

- `lb.c`의 `SERVER_COUNT`와 실제 실행하는 서버 수를 맞춰야 합니다. 서버 수가 부족하면 LB가 `accept()`에서 블로킹됩니다.
- 테스트 환경: Ubuntu, x86-64, VM 4~11대 (LB / 서버 * 2 / 클라이언트 * (1 - 8))
